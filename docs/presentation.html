Scope · v1.0

What ships today, and what stays out (on purpose).

In scope

  • Catalog: AWS Glue Data Catalog
  • Storage: Amazon S3
  • Format: Iceberg format-version 2 (row-level deletes, merge-on-read)
  • Stacks: Glue PySpark, EMR PySpark, ECS Python, Lambda Python, ECS Java, Lambda Java
  • Multi-region: S3 Cross-Region Replication + metadata repointing + per-region Glue registration
  • Tenancy: per-producer profiles with preset inheritance
  • Docs: admin guide, user guide, stack-specific recipes

Out of scope (v1.0)

  • Clouds: Azure, GCP, on-premises
  • Catalogs: Unity Catalog, Polaris, Nessie, Hive Metastore
  • Multi-Region Access Points — intentionally disallowed
  • Cross-region S3 reads at runtime — intentionally disallowed
  • Transactional cross-region writes (eventual consistency only via CRR)
  • SaaS runtime — Iceshelf is a workflow package, not a service

Why these exclusions are deliberate: the customers we’re optimizing for are regulated shops who would reject MRAPs and cross-region S3 in a security review anyway. Every exclusion here simplifies compliance review for the producer.

Resilience · numbers you can put on a risk register

RPO, RTO, and the failure modes we planned for.

Targets are contract-grade defaults, tunable per producer via the profile. Every number below is a knob, not a guess — we generate the CloudWatch alarms to prove them.

Recovery targets (default profile)

MetricTargetDriver
RPO — data loss window≤ 20 minS3 CRR lag p99 + last repointed snapshot age
RTO — time to query in DR≤ 60 minRepointing job + Glue re-register + runbook sign-off
Repointing cadenceevery 15 minScheduled EventBridge → Lambda/Step Functions
CRR replication SLA (AWS)15 min for 99%S3 Replication Time Control (RTC), opt-in per profile

Tighter targets supported — e.g. 5-minute RPO with RTC + 5-minute repointing — at higher CRR cost. The profile makes the tradeoff explicit.

Failure-mode matrix

ScenarioDetectionBehavior
CRR lag spike (> 30 min) CloudWatch alarm on BytesPendingReplication Repointing skips that commit; DR stays on last-known-good snapshot; page on-call.
Repointing job fails mid-rewrite Step Functions task failure + alarm Atomic swap: DR catalog keeps prior metadata_location. No partial state.
Primary-region write during failover Runbook gate: primary write-disable before cutover Writers suspended (IAM deny) → final repointing → DR takes writes. No split-brain.
Stale DR catalog at query time Snapshot-age metric per table Readers see last repointed snapshot; alarm fires if age exceeds RPO budget.
Orphaned data files in DR bucket Monthly orphan-cleanup job Files older than retention + unreferenced are GC’d. Repointing never deletes.
Failback to primary after outage Runbook: reverse CRR + repoint the other way Same utility, flipped source/target. Primary re-registered from DR’s current state.

What we test

  • Quarterly game-day: simulated primary-region outage with stopwatch
  • Chaos tests: repointing job killed mid-run, CRR throttled, Glue register denied
  • DR query correctness: hash-compare row counts & sample partitions across regions

Known constraints

  • CRR is eventually consistent — sub-minute RPO is not achievable without synchronous cross-region writes (prohibited)
  • DDL changes must be re-applied in DR catalog; captured in runbook, not automatic
  • Active-active writes are out of scope; pattern is active-passive by design

Evidence for audit

  • Game-day report template pre-populated from CloudWatch
  • RPO/RTO dashboards per producer + fleet rollup
  • Runbook is version-controlled alongside the pipeline code — one source of truth
Architect Leadership
Security, compliance & governance

Defensible by default — built for security review, not around it.

Identity & access

  • IAM policies scoped to arn:aws:glue:REGION:ACCOUNT:database/<db> and table/<db>/* — never "*"
  • Lake Formation fine-grained access supported via profile (lf_tags)
  • Separate read, write, and maintenance roles generated per producer
  • No cross-account catalog writes by default; opt-in via explicit resource policy

Encryption & key management

  • S3 bucket encryption defaults to aws:kms with a CMK alias pinned in the profile
  • Glue Catalog encryption supported; connection-credential encryption enforced when declared
  • Per-producer CMK isolation possible — one key per data domain is a one-line profile change
  • Key-grant list captured in profile so rotation & audit are scripted, not tribal knowledge

Network isolation

  • VPC endpoints for S3 · Glue · STS · KMS · CloudWatch encoded in generated Terraform
  • Generated code never requires public egress to an Iceshelf-hosted service (there is none)
  • Private-subnet-only deployments supported out of the box
  • Multi-region: every data-path hop stays in-region; CRR is the only inter-region flow

Audit & governance

  • Profile & generated artifacts are diff-reviewable in Git — every change is attributable
  • Required-tags enforced at generation time (e.g. CostCenter, DataClassification, Owner)
  • Optional audit bucket pattern: every Iceberg commit event mirrored for WORM retention
  • Snapshot expiration + orphan cleanup scheduled with retention windows the auditor can read

Control-framework alignment

SOC 2

Least-privilege IAM, change-controlled artifacts, encryption-at-rest defaults, traceable commits.

HIPAA / PHI

CMK per domain, VPC-only network path, Lake Formation row/column access, audit-log mirroring.

PCI-DSS

Tokenization is upstream; Iceshelf preserves strict access scoping and supports segregated key hierarchy.

FedRAMP / IL5-adjacent

No cross-region S3, no MRAPs, no third-party runtime. Artifacts deployable in GovCloud with a profile swap.

Iceshelf is a code generator; it cannot be certified, but its output is designed so that your pipelines pass the controls that matter for each framework. Specific framework attestations remain your responsibility.

Specialized subagents

Five agents. Each with a narrow mandate and the right model tier.

iceberg-orchestrator

Opus

End-to-end coordinator for /iceberg-onboard. Reads profile, plans artifacts, seeks user approval, delegates, aggregates output into a deployment package. Parallelizes independent work.

iceberg-architect

Opus

Reads the producer’s current code and infrastructure. Returns migration strategy, risk matrix, downstream-consumer compatibility assessment. Never touches production directly.

iceberg-code-generator

Sonnet

Generates per-stack production code. Knows PySpark, PyIceberg, and Java idioms. Ships with safe Java writer pattern, Glue Job.init/commit lifecycle, pinned region config.

iceberg-validator

Sonnet

Read-only static checks: schema, Spark config completeness, PyIceberg canonical region keys, IAM scope, Avro file-level metadata preservation in multi-region output.

iceberg-multi-region-planner

Sonnet

S3 CRR Terraform, metadata repointing utility (Python or Java), sync automation via EventBridge + Lambda, monitoring, and a failover runbook — all deployed to the producer’s account.

Isolation model

Every agent starts with zero prior context. The orchestrator passes full context on each spawn. No agent retains memory of prior runs.

Why this matters: reproducibility and reviewability. You can replay any phase from the captured inputs.
